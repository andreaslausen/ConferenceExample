= Architecture Decisions

== Architecture Decision Records

=== ADR-1: Modular Monolith as Deployment Architecture

[cols="1,3",options="header"]
|===
| Attribute | Value

| Status
| Accepted

| Context
| The system consists of multiple bounded contexts (Conference, Talk, Voting, Notification, Attendee, Content). A deployment strategy must be chosen that balances modularity, operational simplicity, and the ability to evolve the architecture over time.

| Decision
| Deploy all bounded contexts as modules within a *single ASP.NET Core application* (modular monolith). Each module owns its own database schema within a shared database instance. Modules communicate in-process via defined interfaces (domain events, mediator, or explicit contracts) — not via network calls.

| Rationale
a|
* *Operational simplicity* — one deployable unit, one database instance. No distributed-systems overhead (service discovery, network resilience, distributed transactions).
* *Developer experience* — straightforward local development and debugging; all modules run in a single process.
* *Strict boundaries* — despite sharing a process, modules are isolated via schema-per-module and enforced dependency rules (architecture tests). This preserves the option to extract modules into separate services if scaling or organisational needs require it.
* *Demonstration value* — shows how DDD bounded contexts can be implemented with strong isolation without prematurely adopting microservices.

| Consequences
a|
* *Positive:* Low operational complexity, fast in-process communication, simple transactions within a module, easy local development.
* *Negative / trade-offs:* All modules must be deployed together; a bug in one module can affect the entire application. Independent scaling of individual modules is not possible without extraction. Discipline is required to maintain module boundaries over time.
|===

=== ADR-2: Event Sourcing and CQRS

[cols="1,3",options="header"]
|===
| Attribute | Value

| Status
| Accepted

| Context
| The system serves as a demonstration project for talks and workshops, as well as a real-world example of modern architectural patterns. For conference and talk management, traceability of changes and efficient queries for different views are relevant concerns.

| Decision
| Adopt *Event Sourcing* combined with *CQRS* (Command Query Responsibility Segregation). Aggregate state is not stored as a current snapshot but as an immutable sequence of domain events. The write and read sides are explicitly separated: commands mutate state via the event store, while queries are served by optimised read models (projections).

| Rationale
a|
* *Demonstration value* — the pattern is well-suited for illustrating DDD, domain events, and Clean Architecture in talks and workshops.
* *Read-side performance* — read models can be tailored exactly to each use case (e.g. conference overview, talk list by track) and are pre-computed, so no joins or aggregations are required at query time.
* *Complete audit trail* — all state changes are persisted as events; no information is ever lost through overwriting.
* *Temporal queries* — the state at any point in the past can be reconstructed by replaying events up to a given position in the stream.
* *Debugging and traceability* — bugs can be reproduced and analysed by inspecting the exact sequence of events that led to a given state.
* *Natural expressiveness in the domain model* — domain events (`TalkSubmitted`, `SpeakerConfirmed`, …) are first-class citizens and directly reflect the ubiquitous language.
* *Independent scaling* — the read and write sides can be scaled separately; read models can be materialised in a query-optimised store (e.g. Redis, Elasticsearch).
* *Integration capability* — the event stream provides a natural foundation for integrating with other systems or bounded contexts via asynchronous message processing.

| Consequences
a|
* *Positive:* High traceability, flexible read models, good testability of aggregates via event sequences.
* *Negative / trade-offs:* Higher initial complexity compared to a classic CRUD approach; eventual consistency between the write and read sides must be consciously accounted for; event schema evolution requires care (upcasting).
|===

=== ADR-3: Frontend Technology — React with Vite and TypeScript

[cols="1,3",options="header"]
|===
| Attribute | Value

| Status
| Accepted

| Context
| The system requires a user interface for conference organizers, speakers, and attendees. A frontend technology must be chosen that supports modern UI patterns (particularly drag-and-drop for schedule management), is widely adopted with strong community support, and integrates well with the REST API exposed by the .NET backend. The frontend should be independently deployable to allow for flexible deployment strategies.

| Decision
| Use *React with Vite and TypeScript* as the frontend stack. The frontend is a separate single-page application (SPA) that consumes the backend via the REST API. Component styling is done with *Tailwind CSS* and *shadcn/ui*, a modern component library built on Radix UI primitives. Drag-and-drop functionality is implemented using *@dnd-kit*, the current standard library for accessible, performant drag-and-drop in React.

| Rationale
a|
* *React ecosystem* — React is the most widely adopted frontend library (18M+ weekly npm downloads), ensuring strong community support, extensive documentation, mature tooling, and a vast ecosystem of libraries and components.
* *TypeScript* — provides type safety and excellent integration with the OpenAPI-generated API client, reducing runtime errors and improving developer experience through autocomplete and refactoring support.
* *Vite* — modern build tool offering extremely fast hot module replacement (HMR) and optimized production builds. Simpler configuration than webpack-based alternatives.
* *shadcn/ui* — rapidly growing component library (trendy and widely adopted, used by Vercel, Supabase, and many production applications). Built on Radix UI primitives (accessibility-first, unstyled components) and styled with Tailwind CSS. Components are copied into the codebase rather than installed as dependencies, allowing full customization.
* *Tailwind CSS* — utility-first CSS framework that has become the industry standard for modern web development. Enables rapid UI development with consistent design tokens.
* *@dnd-kit* — modern, accessible, and performant drag-and-drop library for React. Ideal for building interactive schedule management UIs where talks can be dragged into time slots and rooms.
* *Independent deployment* — the frontend can be deployed separately to static hosting providers (Vercel, Netlify, Cloudflare Pages) or served as static files by the .NET backend. This decoupling enables independent scaling, frontend-specific CI/CD pipelines, and frontend-only updates without backend redeployment.
* *API-first architecture* — the REST API layer is required anyway for public API access and potential future integrations. Using it as the frontend integration layer ensures that the API is well-designed, documented (via OpenAPI), and tested under real load.
* *Blazor rejected* — Blazor Server requires a persistent SignalR connection per user, limiting independent scaling and introducing state management challenges. Blazor WebAssembly has a larger initial payload and longer startup time. Neither offers the mature drag-and-drop ecosystem available in React. Most importantly, the requirement for drag-and-drop and the desire to use widely adopted, JavaScript-based technologies makes React a better fit for this project's evolved goals.
* *Next.js rejected* — while Next.js offers server-side rendering (SSR) and excellent developer experience, it introduces additional complexity (server components, routing conventions, deployment requirements) that is unnecessary for this SPA use case. Vite provides the essential features (fast dev server, optimized builds) without the framework overhead.
* *Vue.js rejected* — while Vue offers a great developer experience and a gentler learning curve, React's larger ecosystem and significantly greater adoption make it the safer choice for long-term maintainability and community support. The drag-and-drop ecosystem in particular is more mature in React.
* *Angular rejected* — Angular's opinionated structure and steeper learning curve introduce more complexity than necessary. React's component-based model and functional programming style align better with modern frontend development practices.

| Consequences
a|
* *Positive:* Modern, widely-supported technology stack; excellent drag-and-drop capabilities for schedule management; independent deployment and scaling; type-safe API integration via TypeScript; strong ecosystem and community support; flexible deployment options.
* *Negative / trade-offs:* Introduces a JavaScript build toolchain (npm, Vite) and a separate frontend codebase; requires maintaining the REST API layer; slightly more complex local development setup (running both backend and frontend); frontend and backend must coordinate on API contracts (mitigated by OpenAPI-generated TypeScript client).
|===

=== ADR-4: Frontend Boundary — Single Shared Application

[cols="1,3",options="header"]
|===
| Attribute | Value

| Status
| Accepted

| Context
| The system has multiple bounded contexts (Conference, Talk, and others planned). A decision is needed on whether the frontend should mirror those boundaries — one frontend application per bounded context (micro frontends) — or whether a single shared application should serve all user-facing functionality.

| Decision
| Use a *single shared React application* that consumes the REST API from all bounded contexts. There is no frontend decomposition along bounded context lines.

| Rationale
a|
* *Demonstration scope* — the bounded context boundaries in this project are an architectural concern of the backend (domain isolation, event stores per context, separate schemas). Replicating those boundaries in the frontend via micro frontends would add complexity without adding demonstration value for the architectural patterns being showcased.
* *Micro frontend trade-offs* — micro frontend architectures introduce significant complexity: independent deployment pipelines per frontend fragment, a shell/composition layer to stitch fragments together, shared dependency management (duplicate React versions, conflicting CSS), cross-fragment state management, and routing coordination. These challenges are worth demonstrating only when frontend autonomy and independent team deployment are explicit requirements.
* *Operational simplicity* — a single React SPA is straightforward to build, deploy, and debug. A unified application bundle simplifies CI/CD, reduces build times, and avoids the overhead of managing multiple frontend projects.
* *User experience* — a shared application provides a unified navigation experience, consistent styling, and seamless transitions between features from different bounded contexts. No shell application or composition layer is needed.
* *API as integration layer* — the REST API already serves as the integration point between frontend and backend. The frontend consumes well-defined API contracts (documented via OpenAPI) and is decoupled from internal backend module boundaries.

| Consequences
a|
* *Positive:* Simple deployment, unified user experience, no composition layer required, easy local development, consistent styling and navigation, single build pipeline.
* *Negative / trade-offs:* The frontend introduces a coupling point across bounded contexts at the UI layer. Features from different contexts are delivered as part of a single deployment unit. If bounded context autonomy at the frontend level were a hard requirement (e.g. independent frontend deployment per context by separate teams), this decision would need to be revisited. For the current scope and goals of this project, that trade-off is explicitly accepted.

| Future Migration Path
a|
If independent frontend deployment per bounded context becomes necessary in the future, React supports several micro frontend architectures:

* *Module Federation* — each bounded context can be extracted into a separate React application that dynamically shares components and dependencies at runtime. Webpack Module Federation (or `@module-federation/enhanced` for Vite) enables this without a full rewrite.
* *Monorepo with independent apps* — each context becomes a separate deployable React app within a monorepo structure (using Nx, Turborepo, or pnpm workspaces). Shared code (components, API clients, types) is maintained in internal packages.
* *Single-spa or similar frameworks* — a shell application orchestrates multiple independently deployed React apps, each owning a portion of the route space.

To facilitate future migration, the codebase should be organized by bounded context from the start (e.g. `src/features/conference/`, `src/features/talk/`) with clear boundaries between context-specific code and truly shared infrastructure. This structure allows individual contexts to be extracted into separate applications with minimal refactoring if and when deployment independence becomes a requirement.
|===

== Open Decisions

=== OD-3: Event Store Persistence Technology

[cols="1,3",options="header"]
|===
| Attribute | Value

| Status
| Open

| Question
| Which persistence technology should be used as the event store?

| Options Considered
a|
* *EventStoreDB* — purpose-built event store with native support for event streams, subscriptions, and projections. Ideal fit for Event Sourcing, but introduces an additional infrastructure component.
* *PostgreSQL* — relational database used as an event store via an append-only events table. Familiar technology, easy to operate, but requires more custom infrastructure code.
* *SQL Server* — same approach as PostgreSQL; already common in .NET ecosystems.
* *MongoDB* — document-oriented store that can serve as an event store via an append-only collection. Designed to handle large volumes of data efficiently, which is relevant given that events are never deleted and the store grows continuously over time. Good .NET support via the official MongoDB driver.
* *Marten* — .NET library that turns PostgreSQL into a document/event store with first-class Event Sourcing support and projections built in.

| Decision Criteria
| Operational simplicity, .NET integration, suitability as a demonstration technology, and alignment with the existing infrastructure.
|===

=== OD-4: Read Model Persistence Technology

[cols="1,3",options="header"]
|===
| Attribute | Value

| Status
| Open

| Question
| Which persistence technology should be used to store and serve read models (projections)?

| Options Considered
a|
* *In-memory store* — projections are held in memory and rebuilt on startup by replaying all events. Zero operational overhead, but not suitable for production deployments with large event streams.
* *PostgreSQL* — projections are materialised into relational tables. Flexible querying via SQL, easy to operate alongside the event store if PostgreSQL is already chosen there.
* *Redis* — in-memory key/value store for very fast reads. Well-suited for simple, flat projections; less ideal for complex query patterns.
* *Elasticsearch / OpenSearch* — full-text search and complex aggregations out of the box. High operational overhead; most valuable when rich search capabilities are required.
* *MongoDB* — document-oriented store well-suited for read models, as projections map naturally to documents. Good .NET support via the official MongoDB driver; flexible schema makes it easy to evolve projection shapes without migrations.
* *Marten (with PostgreSQL)* — if Marten is chosen for the event store, it also provides built-in projection support writing into PostgreSQL, avoiding a separate read-side store entirely.

| Decision Criteria
| Query complexity, read performance requirements, operational simplicity, and consistency with the chosen event store technology.
|===

=== OD-5: Identity Provider

[cols="1,3",options="header"]
|===
| Attribute | Value

| Status
| Open

| Question
| Which identity provider should be used for authentication and role management?

| Options Considered
a|
* *Keycloak* — open-source, self-hosted, feature-rich (SSO, social login, fine-grained roles). Requires operational effort for hosting and maintenance.
* *Auth0* — managed SaaS identity platform. Low operational effort, but introduces a vendor dependency and recurring costs.
* *Microsoft Entra ID (formerly Azure AD)* — managed identity service, strong integration with the Microsoft/.NET ecosystem. Ties the system to the Azure platform.

| Decision Criteria
| Operational simplicity (self-hosted vs. managed), cost, alignment with the .NET ecosystem, ease of local development (Docker-based setup), and suitability as a demonstration technology.
|===
